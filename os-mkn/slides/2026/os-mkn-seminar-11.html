Базовая подготовка

  • Отключаем процессы по максимуму
  • Оставляем самое необходимое - без чего совсем никак
  • Или нересурсоемкое
  • Обеспечиваем наличие значительного свободного места на диске
  • Случайные обстоятельства сглаживаем повторяющими экспериментами
  • Оцениваем дисперсию

for i in 0..1000 {

Проблемы

  • Файловый дескриптор неплохо бы закрывать
  • Но закрытие само по себе не означает запись данных на диск
  • Нужно позаботиться о явном синке
  • Иначе получим гипероптимистичный результат

Проблемы

  • Данных записывается очень мало
  • Время на позиционирование будет доминировать над временем на запись
  • И может примешиваться время на работу с метаданными файла
  • Сопоставимое с временем на записьо
  • Лучще писать мегабайтами
  • Или десятками/сотнями

Проблемы

  • Конкретные данные тоже важны
  • Для файловых систем семейства ext важно, чтобы были не нули
  • Блоки только из нулей не хранятся физически на диске
  • Хранение ограничивается маркерами в метаданных
  • Рискуем получить гипероптимистичный результат

Чтение

  • Нужно учесть опыт записи - в переложении для чтения
  • Но это НЕ решит всех проблем
  • Основная проблема - кеширование
  • Разберем два случая

Чтение

  • Можем читать предзаписанные данные
  • То, что существовало, до каких-либо экспериментов
  • Первое чтение будет реальным
  • Повторные чтения с большой вероятностью будут из кеша

Чтение

  • Можем читать то, что записали в ходе измерения скорости записи
  • И сначала что-то записать в рамках эксперимента по замеру скорости чтения
  • И тут же начать читать
  • Понятно, что нужен fsync
  • Менее очевидно, что fsync сам по себе не спасет от кеширования

Кеш в Linux

  • В самом общем случае есть два пути обойти использование кеша
  • Первый способ - уменьшить его размер вплоть до 0
  • Второй способ - отказаться от его использования в конкретной операции
  • Первый способ в Linux принципиально невозможен
  • Кеш в Linux заполняет весь возможный объем памяти

Память в Linux

  • Есть внутренние структуры ядра
  • Есть память ядра, выделяемая на каждый процесс
  • Это все живет в RAM
  • И вычитается из общего бюджета RAM
  • За все остальное конкурируют три подсистемы

Память в Linux

  • Первая - кеширование внещних устройств
  • Вторая - memory-mapped files
  • mmap-files - файлы, отображаемые на память
  • Третья - swap

Пример


int
main(int argc, char *argv[]) {
    int fd = open(argv[1], O_RDONLY);
    int offset = atoi(argv[2]);
    int length = atoi(argv[3]);
    void *addr = mmap(NULL, length, PROT_READ,
                      MAP_PRIVATE, fd, offset);
    s = write(0, addr, length);
}

Память в Linux

  • Сценарий 1: мало процессов
  • Кто-то интенсивно читает данные с диска
  • Но хранит только небольшую часть
  • Свободная память будет отводиться под кеширование данных с диска
  • Вплоть до заполнения всей памяти

Память в Linux

  • Сценарий 2: 500 процессов
  • Каждый использует примерно по 100M
  • Дисковые операции незначительны
  • Включен swap
  • Суммарно нужно 50G
  • Редко используемые страницы будут вытесняться на диск

Память в Linux

  • Сценарий 3: 50 активных процессов
  • Каждый использует примерно по 100M
  • И половина ходит на диск
  • В какой-то момент память будет занято смесью кеша и памятью процессов
  • Очередное дисковое чтение или подкачка страницы будут приводить к аытеснению
  • И жертва будем искать среди страниц данных и кешированных страниц
  • На общих основаниях

Борьба с кешированием

  • При открытии файла указываем опцию O_DIRECT
  • Обеспечивает чтение напрямую с диска
  • Полезна не только для измерения скорости
  • Пример применения: свое кеширование для СУБД и т.п.
  • Во избежание двойного кеширования
  • И для лучшей управляемости
  • (Но swap может интересно влиять)

Измерение скорости позиционирования

  • По-хорошему надо отдельно мерить позиционирование на чтение и на запись
  • Потому что приготовления могут быть разными
  • В любом случае просто вызывать lseek недостаточно
  • Сам по себе lseek ничего не делает с диском

Измерение скорости позиционирования

  • Нужно позиционироваться и сделать операцию ввода-вывода
  • Например, чтение
  • Снова может помешать кеширование
  • Причем "в обе стороны"
  • Случившееся кеширование может отменить необходимость чтения
  • И заодно - позиционирования

Измерение скорости позиционирования

  • Другой риск - если данный блок не закеширован
  • Когда кеширование есть, но конкретный блок не закеширован - может случиься prefetching
  • prefetching - чтение бОльшего объема, чем требуется для конкретной read-операции
  • Эвристическая оптимизация
  • В нашем случае помешает точности расчетов
  • O_DIRECT решит обе проблемы

Измерение скорости позиционирования

  • Для одиночного замера нужно спозиционироваться через lseek
  • И выполнить чтение заданного размера - в режиме O_DIRECT
  • И из замеренного времени вычесть время на чтение данных
  • Которое мы уже получили более или менее
