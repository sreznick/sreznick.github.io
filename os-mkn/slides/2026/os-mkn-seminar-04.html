Первый вход в C

  • Обнуляем секцию bss
  • Выделяем место на стек вызовов
  • (Возможно, что временно)
  • Устанавливаем регистры RSP/RBP
  • Если точка входа предполагает параметры - записываем их в соответствии с соглашением о связях
  • Переходим на точку входа - через CALL или JMP
  • В данном случае JMP - допустимый вариант

Вспомогательные функции

  • Пишем код вспомогательных ассемблер функций
  • Соблюдаем соглашение о связях
  • При вызове C-функций из ассемблер кода соблюдаем соглашения о связях
  • Есть еще вариант интеграции ассемблер фрагментов в C
void read_idtr(struct idtr *idt_reg) {
    __asm__ __volatile__(
        "sidt %0\n"
        : "=m" (*idt_reg)
        : // no input
        : // no clobber
    );
}

QEMU

  • Но код для исполнения проще порождать в WSL или на виртуалке

QEMU

  • Попытка запуска: qemu-system-x86_64
  • (Похоже на запуск ПК без диска)
  • Можно так: qemu-system-x86_64 -monitor stdio

  • В замороженном состоянии: qemu-system-x86_64 -S -monitor stdio

Старт реального железа и старт QEMU

  • Часть адресного пространства процессора отображается на ROM
  • Эта часть включает адрес, с которого начинается исполнение при старте
  • В ноутбуках этот код делает производитель/сборщик
  • Разработчики ОС в него обычно не вмешиваются
  • ОС вступает в дело чуть позже
  • Ключевые слова этого этапа: BIOS, UEFI

Старт реального железа и старт QEMU

  • Стартовый код проводит базовое тестирование
  • Дает возможность зайти в базовые настройки: указать загрузочный диск и т.п.
  • Если не свернули в особые ветки - стартовый код начинает перебирать загрузочные устройства
  • Порядок перебора может регулироваться базовыми настройками
  • На устройстве проверяется наличие загрузочного сектора

Старт реального железа и старт QEMU

  • Если нашли - он загружается в память
  • (Стартовый код должен уметь общаться с диском без помощи ОС)
  • В загрузочном секторе может быть логика загрузки ОС
  • Особенно если ОС - единственная на данной машинке
  • И это пользовательская машинка
  • Но иногда хочется иметь несколько вариантов ОС
  • Особенно если кто-то любит поиграть со сборкой ядра

Загрузчик

  • Вспомогательное приложение, состоящее из двух частей
  • Одна часть работает как приложение в рамках ОС
  • Устанавливает вторую часть - работающую вне ОС
  • Вторая часть загружается с загрузочного сектора
  • Предлагает меню вариантов загрузки
  • Среди вариантов могут быть загрузочные сектора разных дисков
  • Но не только

Загрузчик

  • Альтернативный вариант - файл на диске
  • При выборе этого варианта содержимое файла загружается в память
  • Формат файла может напоминать обычный исполняемый файл
  • С небольшими отличиями
  • Например, не нужно иметь точку входа main
  • Это упрощает линковку

QEMU

  • Технически QEMU позволяет проимитировать загрузку, начиная с кода из ROM
  • Это усложняет путь
  • И это усложнение - не совсем про операционные системы
  • QEMU дает возможность проскочить сразу к работе загрузчика
  • В случае x86 - подготовить файл с образом ядра ОС
  • И указывать его при старте QEMU

Как отлаживать

  • QEMU не дает возможности пошагового исполнения сам по себе
  • Можно через GDB - но это еще отдельно разбираться
  • Можно сразу научиться печатать
  • При старте получаем доступ к видеопамяти
  • Текстовый режим
  • Экран 80x25
  • По 2 байта на позицию

Простой код для железки

  • В ассемблере нужно отдельное приседание ради загрузки
  • Порожденный файл должен быть узнан загрузчиком, работающим в qemu/железке
  • Чтобы сработал протокол Multiboot
  • Нужно прописать некоторые магические значения

Пример

  global start                    ; the entry symbol for ELF
    extern entry

    MAGIC_NUMBER equ 0x1BADB002     ; define the magic number constant
    FLAGS        equ 0x0            ; multiboot flags
    CHECKSUM     equ -MAGIC_NUMBER  ; calculate the checksum
                                    ; (magic number + checksum + flags should equal 0)

    section .text:                  ; start of the text (code) section
    align 4                         ; the code must be 4 byte aligned
        dd MAGIC_NUMBER             ; write the magic number to the machine code,
        dd FLAGS                    ; the flags,
        dd CHECKSUM                 ; and the checksum

    start:                          ; the loader label (defined as entry point in linker script)

Очистим экран в ассемблере

    start:

      mov ebx, 0xb8000 ; VGA area base
