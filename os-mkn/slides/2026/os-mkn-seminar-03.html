Линковка

  • Во время линковки должны быть найдены все локально неразрешенные символы
  • И сформирован единый исполняемый файл
  • Детали его формирования могут зависеть от целевой архитектуры
  • И от конкретных пожеланий пользователя
  • Имеет значение - используется ли "плоская" модель памяти или страничная организация
  • Для плоской модели имеет значение, в каких адресах хотим разместить код/данные
  • Для этого используется линкерный скрипт - даже если вы этого не осознаете

Линковка

  • В каждой конфигурации GCC есть свой вариант скрипта по умолчанию
  • Его можно увидеть: ld --verbose
  • Его обычно не меняют при обычной разработке
  • Его часто нужно создавать при специфичной конфигурации
  • Например, для работы на чистом железе без ОС

Понятия линковки

  • Есть n входных файлов - "object files"
  • Есть 1 выходной файл - тоже "object"
  • Иногда выходной файл называют "executable"
  • Но не всегда порожденный файл предназначен непосредственно исполняться
  • Например, это может быть динамическая библиотека

Понятия линковки

  • Объектный файл состоит из секций/section
  • У секций есть имя, содержимое и размер
  • Основные секции - .text, .data, .bss
  • В .text по умолчанию попадает порожденный компилятором код для C-исходников
  • В .data по умолчанию попадают глобальные переменные из C-кода
  • В .bss по умолчанию попадают статические переменные из C-кода
  • GCC и ассемблеры позволяют вручную направлять элементы данных/кода в любые секции

Понятия линковки

  • Секция может быть загружаемой - loadable
  • Содержимое такой секции загружается в память при исполнении
  • .text и .data - загружаемые секции
  • Секция может быть размещаемой - allocatable
  • Под такую секцию выделяется память при исполнении
  • .bss - размещаемая секция
  • Ее содержимое заполняется нулями при старте

Понятия линковки

  • С объектным файлом связан набор символов
  • Символ - пара (имя, адрес)
  • Компилируемый C-код порождает символы из объявленных функций/переменных (глобальных и статических)
  • Ассемблер порождает символы из меток, объявленных внешними
  • Линкерный скрипт может породить свои символы
  • Объектные файлы могут содержать неразрешенные (non-resolved) символы
  • Однозначное разрешение символов - необходимое условие успешной линковки

Пример


     SECTIONS
     {
       . = 0x10000;
       .text : { *(.text) }
       . = 0x8000000;
       .data : { *(.data) }
       bss_begin = .
       .bss : { *(.bss) }
       bss_end = .
     }

Регистры x86

  • Современный вариант включает несколько слоев обратной совместимости - отражая историю развития
  • 16 регистров общего назначения
  • 8 более исторических в CISC-духе
  • 8 свежих в RISC-духе: R8 - R15
  • Каждый исторический имеет 3-4 суженных проекции - доступные как отдельные регистры
  • 64-битные регистры: RAX, RBX, RCX, RDX, RSI, RDI, RSP, RBP
  • 32-битные проекции: EAX, EBX, ECX, EDX, ESI, EDI, ESP, EBP
  • 16-битные проекции: AX, BX, CX, DX, SI, DI, SP, BP
  • 8-битные (младшие) проекции: AL, BL, CL, DL, SIL, DIL, SPL, BPL
  • 8-битные (старшие) проекции: AH, BH, CH, DH

Регистры x86

  • RAX - accumulator
  • RBX - base
  • RCX - counter
  • RDX - data
  • RSI - source
  • RDI - destination
  • RSP - stack pointer
  • RBP - stack base pointer

Регистры x86

  • RIP/EIP/IP - instruction pointer
  • CS/DS/ES/SS/FS/GS - сегментные регистры (ограниченного применения)
  • RFLAGS/EFLAGS - флаги
  • CR0-CR8, XCR0 - управляющие
  • DR0-DR7 - отладочные
  • GDTR/LDTR/IDTR - регистры защищенного режима
  • Регистры для "математических" расширений

Типы инструкций x86

  • Операции: арифметические, логические, сравнения
  • Передача данных в регистры и из регистров
  • Передача управления
  • Строковые инструкции
  • Работа со спецрегистрами
  • Прочее

Инструкции-операции в x86

  • Принцип двух аргументов: результат совпадает с одним из операндов
  • Один из операндов - регистр
  • И результат - регистр
  • Второй операнд - регистр, непосредственное значение или адрес в памяти
  • Операция меняет некоторые биты в регистре флагов
  • Это используется последующей опереции условного перехода по состоянию флага
  • Пример: JZ после SUB

Передача данных

  • Непосредственный аргумент в регистр: mov rax, 123
  • Между регистрами: mov rax, rbx
  • В регистр из непосредственного адреса: mov rax, [0x1234]
  • В регистр из непосредственного адреса: mov rax, [variable]
  • В регистр через косвенную адресацию: mov rax, [rbx]
  • В регистр через косвенную адресацию со смещением: mov rax, [rbx + 4]
  • В регистр через индексный регистр со смещением: mov rax, [rsi + arrayPtr]
  • В регистр через индексный регистр с базой: mov rax, [rsi + rbx]
  • В регистр через индексный регистр с базой и масштабированием: mov rax, [rsi * 4 + rbx]
  • В регистр относительно RIP: mov rax, [rip + offset]

Поток инструкций

  • При включении питания или рестарте в регистр RIP попадает значение 0xFFFFFFF0
  • Системная плата должна быть сконфигурирована так, чтобы это был адрес ПЗУ
  • Начинается исполнение потока инструкций
  • Большинство инструкций предполагают последовательное исполнение
  • И неявное увеличение содержимого регистра RIP
  • Есть инструкции перехода: прямого и косвенного, условного и безусловного
  • Большая группа инструкций начинается с J - просто меняющие знания RIP

Поток инструкций

  • Есть инструкции вызова и возврата: CALL/RET
  • Они оперируют с понятием стека
  • Стек живет в памяти
  • За него отвечают 2 регистра: RSP и RBP
  • RSP - stack pointer
  • RBP - base pointer
  • Почему два регистра для организации стека ?

Вызовы и стек

  • Стек нужен для хранения адреса возврата
  • А еще - для передачи аргументов и для хранения локальных переменных
  • Есть пара инструкций PUSH/POP - они двигают RSP
  • CALL/RET тоже двигают RSP
  • Перед вызовом функции может потребоваться исполнение PUSH - чтобы добавить на стек параметр предстоящего вызова
  • Но еще нужно обращаться к локальным переменным на стеке
  • Отталкиваться для этого от RSP неудобно - он может меняться
  • А еще при возврате нужно сбросить со стека все локальные переменные

Вызовы и стек

  • RBP указывает на текущий stack frame
  • Меняется "скачкообразно" при вызове и возврате
  • RSP меняется при подготовке/обратке вызова/возврата
  • В общей схеме возможны вариации
  • Детали зависят от "call conventions"/"соглашения о связях"
  • Соглашения о связях позволяют линковать код от разных компиляторов одного языка
  • Или код на разных языках
  • Или код на ассемблере с кодом языка высокого уровня
