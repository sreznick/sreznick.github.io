xv6

  • xv6 работает на RISC-V
  • toolchain готов
  • docker pull sreznick/xv6-mkn
  • docker run -it sreznick/xv6-mkn
  • Изнутри контейнера: make qemu
  • Должны увидеть: init: starting sh
  • И командную строку
  • Выход: Ctrl-A сначала, потом X (отпустив Ctrl-A)

xv6: что есть

  • Изнутри видна файловая система: ls -l
  • Около 20 файлов
  • Каталогов сходу нет, но можно создать: mkdir d
  • Видны файлы с кодом приложений
  • Набор ограниченный
  • Возможности каждого ограничены

xv6: что есть

  • Почти все файлы в начальном состоянии - приложения
  • Один текстовый - README
  • Часть приложений - тесты
  • Часть - очень упрощенные варианты стандартных Unix-утилит
  • Командная строка - очень упрощенный shell
  • В нем есть конвейер и простые перенаправления ввода-вывода
  • Пример: grep ab README | wc > q

xv6: что есть

  • Готовые приложения не поддерживают --help
  • man-страниц тоже нет
  • Источник знаний - исходники
  • Есть имитация диска, фвйловая система
  • Можно добавить новый файл в начальное состояние системы
  • Нет простого способа добраться до текушего состояния файловой системы с хоста
  • Приложения запускаются из собранных файлов
  • Набор файлов фиксируется на момент сборки

  • Пропишем head везде, где прописан wc
  • Предварительно исполнив make clean
  • Должны получить приложение head с логикой wc

Системные вызовы, обеспечивающие жизненный цикл процесса

  • fork - создание процесса
  • exec - замена текущего приложения на другое
  • wait - ожидание завершения ребенка
  • exit - завершение процесса

Локи

  • Механизм обеспечения целостности
  • CAS - базовый подход к реализации
  • Реентерабельность локов - способность захватить уже захваченный лок
  • И считать количество освобождений
  • Read/Write-локи

Дедлоки

  • Состояние, в котором процессы ждут друг друга
  • И никогда не дождутся
  • Потому что сложился цикл ожиданий
  • Простейший пример:
    • Один процесс захватил lockA и хочет захватить lockB
    • Другой процесс захватил lockB и хочет захватить lockA

Дедлоки

  • Один их способов борьбы с дедлоками - определение дедлока с уничтожением элемента в цепочке зависимостей
  • Это работает в базах данных (отказ в исполнении запроса + ретрай)
  • Но плохо ложится на ОС - удаление процесса сильно болезненнее, чем отказ в исполнении запроса
  • Нужно следить за порядком захвата
  • Обязательное условие сдачи домашки - прохождение стандартных тестов
  • Зависание теста - почти всегда признак дедлока
  • Просто убрать лок - не всегда засчитывается за решение проблемы
