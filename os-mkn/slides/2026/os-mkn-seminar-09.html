Pipe

  • pipe - байтовый поток, не имеющий "внешней" идентификации
  • Создается системным вызовом pipe
  • Процесс может создать пару писатель-читатель над неким буфером
  • Который выделяется ядром и "напрямую" недоступен

Пример


#include 
#include 

int
main(int argc, char *argv[])
{
       int pipefd[2];

       if (pipe(pipefd) == -1) {
           perror("pipe create error");
       }

       printf("r: %d, w: %d\n", pipefd[0], pipefd[1]);
}

Pipe

  • Попытка прочитать из пустого pipe-а завершится ожидание, пока кто-то что-то прочитает
  • Размер pipe-а конечен и фиксирован
  • Если пишем в pipe, из которого не читают или читают медленно - можно перейти в состояние ожидания
  • Пока кто-то что-то не прочитает

Pipe

  • Странно пользоваться pipe-ом внутри одного процесса
  • Типичные сценарии использования предполагают как минимум fork
  • Часто в сочетании с exec
  • Иногда с close/dup

Пример


       int pipefd[2];
       pipe(pipefd);
       for (int i = 0; i < 100000; i++) {
           if (i % 1000 == 0) {
                printf("i: %d\n", i);
           }
           int res = write(pipefd[1], "a", 1);
       }

Сценарии использования

  • От родителя к ребенку
  • От ребенка к родителю
  • От одного ребенка к другому ребенку
  • В общем случае от кого угодно в поддереве, начинающемся от создателя до кого угодно в нем же

Пример


    int pid = fork();

    if (pid == 0) {
        printf("child: %d\n", getpid());
        for (int i = 0; i < 50; i++) {
            char c;
            read(pipefd[0], &c, 1);
            printf("%d: %d\n", getpid(), (int)c);
        }
    }

Пример


    if (pid > 0) {
        printf("parent: %d\n", getpid());

        for (char c = 0; c < 50; c++) {
            write(pipefd[1], &c, 1);
        }
    }

Жизнь после fork-а

  • После fork дескрипторы продолжают самостоятельное существование в обоих процессах
  • Унаследованный ребенком дескриптор может быть унаследован внуком и так далее
  • В момент передачи данных процесс участник коммуникации может не быть предком создателя канала
  • (Если его непосредственный родитель завершил свое существование)

Свойства канала

  • Асинхронный в том смысле, что есть буфер
  • Записать можно, не дожидаясь читателя
  • Почитать можно то, что было записано ранее

Свойства канала

  • Синхронный в том смысле, что при чтении пустого буфера по умолчанию ждем записи
  • Если в начале чтения есть байты, но меньше, чем просили - прочитаем то, что есть
  • Синхронный в том смысле, что при записи в полный буфер по умолчанию ждем чтение
  • Если в начале чтения есть свободные байты, но меньше, чем нужно - запишем то, что есть

Свойства канала

  • Технически любой процесс, имеющий доступ к дескриптору канала, может участвовать в чтении/записи
  • Но осмысленные сценарии с несколькими участниками на одной стороне ограничены
  • Канал - байтовая очередь
  • Пишем всегда в "голову"
  • Читаем всегда из "хвоста"
  • Голова и хвост - свойства канала, а не дескриптора

Пример


    for (int i = 0; i < 5; i++) {
        pid = fork();

        if (pid == 0) break;
    }

Конвейер в bash

  • Синтаксически разбираем командную строку
  • Выявляем участников конвейера
  • По форку на каждого
  • На каждую пару в последовательности создаем pipe
  • До форка

Конвейер в bash

  • После форка в детских ветках закрываем ненужные дескрипторы
  • Еще до exec
  • Отставляем только то, что нужно каждому процессу
  • Дескриптор для чтения и дескритор для записи для тех, кто в середине
  • После этого - exec
  • Но есть нюанс

Конвейер в bash

  • Нужно, чтобы созданные дескрипторы были под номерами 0 и 1 для каждого запущенного процесса
  • А они у нас будут пронумерованы в контексте родительского процесса
  • У которого номера 0 и 1 заняты под свои нужды
  • Нас спасет системный вызов dup

Конвейер в bash

  • Точнее - dup + close
  • dup создает копию дескриптора
  • Принцип/гарантия Unix-систем: при создании дескриптора всегда используется минимальный свободный номер
  • У ребенка есть дескрипторы 0, 1 и 2
  • Они наследованы от родителя
  • Но зачем ему родительские stdin/stdout/stderr ?

Конвейер в bash

  • Закроем их
  • Вызовем dup на pipe-дескрипторами
  • Закроем оригинальные pipe-дескриптора
  • Можно делать exec
  • (Важно dup в правильном порядке вызывать)
