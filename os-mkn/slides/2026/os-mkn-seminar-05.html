Отладка

  • Есть отличия в варианте hosted и free-standing
  • Отличия в технической стороне: как именно ставить точки останова, собирать данные о состоянии и т.п.
  • Значительная часть логики отладка универсальна и платформенно-независима
  • Поэтому есть универсальное средство - gdb
  • Проще научиться основам в хост-варианте

gdb

  • Пишем простое приложение
  • Собираем с опцией -g
  • Запускаем под отладчиком: gdb app
  • Получаем REPL-диалог
  • Готовы звпустить
  • run - запуск
  • Но это фактически без отладки

gdb

  • Есть полные версии команд отладчика
  • run, continue, info registers, step, backtrace, list и т.п.
  • Если короткие версии: r, c, i r, s, bt, l
  • Полезно приучиться к ним - очень ускоряет работу

gdb

  • У многих команд есть параметры
  • Параметры тоже имеют сокращенную форму
  • Радикальная форма сокращения - отсутствие параметров
  • У некоторых команд вызов без параметров приводит не фиксированному поведению
  • А к логическому продолжению прошлых команд

gdb

  • Пример: команда list
  • Показывает несколько строк исходников
  • Первое исполнение без параметров показывает текущее место в коде
  • Можем параметром указать место показа
  • Например, имя функции
  • Или имя исходного файла и номер строки
  • Последующие исполнения без параметров будут показывать следующие строки

gdb

  • breakpoint/b - точка останова
  • Задаем в терминах исходников или адресов
  • Варианты задания в терминах исходников пересекаются с вариантами параметра команды list
  • И других команд работы с исходниками
  • На входе в функцию func: b func
  • На строке 15 файла f.c: b f.c:15

gdb

  • run/r - запуск приложения
  • Аргументы run - аргументы командной строки приложения
  • Проходя через точку останова - останавливаемся
  • Классический подход к реализации точки останова - специальная инструкция
  • Отладчик записывает ее на место обычной инструкции
  • При исполнении срабатывает прерывание/ловушка
  • Управление передается отладчику

gdb

  • В точке останова можем смотреть состояние
  • На разных уровнях и с разных ракурсов
  • print/p - печать значений переменных
  • Но это не просто печать
  • Это исполнение выражений в текущем контексте
  • Печать значения переменной v: p v
  • Печать значения переменной v с инкрементом: p v++
  • Вызов функции f: p f()

gdb

  • x - дамп памяти
  • Указывать адрес
  • После x через / указываются модификаторы
  • Буквенные указывают на интерпретацию содержимого
  • Численные указывают на количество элементов
  • x/10x 0x10000 # 10 16-ричных чисел, начиная с адреса 0x10000
  • x/10i $pc # 10 инструкций, начиная с адреса, равного содеожимому $pc

gdb

  • step/s, next/n - пошаговое исполение
  • С заходом в функции или без захода
  • По строкам исходников или по инструкциям
  • backtrace/bt - просмотр стека вызовов
  • frame - переход на фрейм в стеке вызовов
  • info информация о разных аспектах текущего состояния
  • info registers / i r - состояние регистров

Удаленная отладка

  • Удаленная отладка - код работает где-то на другом устройстве
  • Hа устройстве работает агент
  • Отладчик взаимодействует с ним
  • Манипуляции с устройстовом - через агента
  • Интерпретация состояния - в точке работы отладчика

Удаленная отладка с QEMU

  • В QEMU встроен gdb-агент, нужно только включить опцией -s <номер порта>
  • И заморозить исполнение опцией -S
  • На стороне gdb: target remote localhost:<номер порта>
  • Особенность x86 real mode: адрес инструкции формируется как $cs * 16 + $eip
  • gdb не умеет нормально интерпретировать real mode
  • В целом работает, но очень адекватно показывает состояние

xv6

  • xv6 работает на RISC-V
  • toolchain готов
  • docker pull sreznick/xv6-mkn
  • docker run -it sreznick/xv6-mkn
  • Изнутри контейнера: make qemu
  • Должны увидеть: init: starting sh
  • И командную строку
  • Выход: Ctrl-A сначала, потом X (отпустив Ctrl-A)
