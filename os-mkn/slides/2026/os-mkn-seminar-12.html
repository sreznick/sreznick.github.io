Управление памятью

  • Прооисходит на нескольких уровнях
  • На уровне ядра и на уровне приложения
  • Самое сложное - управление на уровне приложения
  • Запросы могут быть произвольного размера
  • Включая очень маленькие и не степени двойки
  • И нельзя использовать преобразование адресных пространств

Управление памятью

  • Приложение запрашивает память у библиотеки через malloc/new
  • И освобождает через free/delete
  • Освобожденная память может не отдаваться ядру
  • И переиспользоваться библиотекой
  • Если не хватает - библиотека просит у ядра

Управление памятью

  • Запросы на добавление памяти к куче могут быть разного размера
  • Но они кратны странице
  • В адресном пространстве процесса они занимают сплошную область
  • Но не обязаны быть сплошными в физической памяти
  • Аналогично с кодом и статическими/глобальными данными

Запросы внутри ядра

  • Внутри ядра нужна память под свои структуры
  • Например, таблица страниц для каждого процесса
  • Локи
  • Структуры процессов
  • Cтруктуры файлов

Запросы внутри ядра

  • Запрос может быть на разные размеры
  • Для структур ядра не можем использовать преобразование адресных пространств
  • Начнем со случая 4K и больше
  • Ограничимся размерами, равными степени двойки блоков по 4K

Блоки разных размеров

  • Заведем отдельную "подсистему" для выделения блоков каждого размера
  • 4K, 8K, 16K, ...
  • Но не каждого статического размера
  • Будем при необходимости "заимствовать" блоки из пула бОльшего размера
  • И стараться собирать из освобожденных блоков меньшего размера свободные блоки бОльшего размера

Блоки разных размеров

  • В начале имеет по 1 блоку разных степеней двойки
  • Возможно, не каждого из поддерживаемого
  • Зависит от размера свободной памяти
  • Пример: доступно в целом 320M
  • Один блок 256M и один блок 64M

Выделение блока

  • При запроск ищет блок ровно заказанного размера
  • Если есть - выделяем
  • Если нет - ищет блок следующего размера
  • Если есть - выделяем, делим пополам, ненужная половина остается свободной
  • Если нет - рекурсивно повторяем процесс

Освобождение блока

  • При освобождении блока смотрим на соседей
  • Если есть свободный сосед того же размера - попробуем объединить
  • И рекурсивно повторим попытку
  • Пока получается объединять

Представление данных: варианты

  • Есть альтернативы по разным аспектам
  • Можно базироваться на связных списках свободных и занятвх блоков
  • Можно вместо списков использовать битовые массивы
  • Можно прямо или косвенно представлять структуру блоков деревьями
  • Можно хранить заговолки рядом с выделяемой памятью
  • Можно рядом держать управляющие данные

Вариант реализации

  • Для каждого размера - битовый массив свободных блоков
  • По адресу блока и адресу начала области памяти можем понять номер блока данного размера
  • По номеру блока можем понять позицию в дереве блоков - не строя само дерево
  • По позиции в дереве блоков можем понять номер брата - и его состояние

slab-аллокатор: мотивация

  • На уровне ядра могут требоваться маленькие области памяти
  • Например, для представления процесса
  • Метод близнецов плохо подходит для маленький структур
  • Можно воспользоваться тем, что выделяемая память - не совсем произвольного размера

slab-аллокатор: мотивация

  • В контексте ядра память выделяется на несколько типов структур 
  • У каждой - свой размер
  • Организуем для каждого типа свой пул
  • В рамках одной страницы будем выделять n структур
  • И переиспользовать освобожденные
