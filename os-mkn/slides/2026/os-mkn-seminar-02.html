Компилятор

  • Строго говоря - порождает машинные инструкции или объектный код
  • Часто то, что воспринимается как компилятор, заодно вызывает линковку
  • Но технически это отдельный инструмент
  • C-компилятор компилирует каждый файл отдельно
  • Для внешних символов (объявленных, но не определенных идентификаторов/функций) порождаются заглушки
  • Популярный компилятор - GCC

Компилятор

  • Можно ограничиться порождением ассемблера: gcc -S f.c
  • Можно ограничиться компиляцией без линковки: gcc -c f.c
  • Можно дизассемблировать объектный файл: objdump -S f.o
  • objdump - часть binutils
  • GCC поддерживает кросс-компиляцию
  • Но не в рамках одной утилиты
  • Из одной кодовой базы собираются/устанавливаются разные конфигурации компилятора

Компиляторы

  • Установка кросс-компилятора для riscv: apt install gcc-riscv64-unknown-elf
  • Установка кросс-компилятора для i686: apt install gcc-i686-linux-gnu
  • После установки: i686-linux-gnu-gcc, riscv64-unknown-elf-gcc
  • Заодно установятся binutils: i686-linux-gnu-objdump, riscv64-unknown-elf-objdump
  • Компилятор не использует все машинные инструкции - для них нет семантических эквивалентов в языках программирования
  • В некоторых ситуациях не обойтись без программирования на ассемблере

Ассемблер

  • Программирование в явно выписанных машинных инструкциях
  • В символьной форме
  • Структура ассемблер-кода для разных архитектур похожа
  • Отличаются инструкции, структура их аргументов и набор регистров
  • Это дает идейную основу для универсального ассемблера

Ассемблер

  • Производители процессоров заинтересованы в наличии ассемблера для своего процессера
  • И часто создают свои ассемблеры - только для своей архитектуры
  • Бывают независимые разработчики ассемблеров для одной архитектуры
  • Или поддерживаются производителями, или сами делают что-то свое по своим бизнес-планом
  • И фокусируются на отдельной архитектуре

Ассемблер

  • Универсальный ассемблер - gas (GNU assembler)
  • Устанавливается через apt вместе с компилятором
  • Принимает код, который был порожден GNU-компилятором
  • objdump порождает код по правилам GNU-ассемблера
  • У него есть много платформенно-независимой логики: макросы и т.п.
  • А минусы будут ?

Ассемблер

  • Субъективно gas для x86 хуже отличается по стилю привычных форм x86-ассемблера
  • В частности - от примеров в документации Intel
  • Более объективно - в записи операций gas использует ATT-нотацию
  • А в других ассемблерах - обычно используется Intel-нотация
  • ATT: source, dest
  • Intel: dest, source

Ассемблер

  • Для чтения того, что породил GCC, использование GAS неизбежно
  • Для работы с RISC-V - тоже
  • Для Intel-ассемблирования привлекателен вариант альтернативного ассемблера, работающего в Intel-нотации
  • Живой пример такого ассемблера - nasm

Типы реализации C

  • Реализация C бывает двух типов: hosted и free-standing
  • hosted предполагает наличие операционной системы там, где будет исполняться приложение
  • free-standing предполагает исполнение там, где есть просто память и процессор
  • hosted должна знать конкретную ОС
  • Поэтому в GCC есть два параметра в настройке инструментов: target CPU и target OS
  • hosted организует связь с загрузчиком и вызов функции main
  • hosted рассчитывает на системные вызовы и сводит к ним многие библиотечные функции
  • free-standing ни на что не рассчитывает
  • И предоставляет маленький поднабор библитечных функций - для которых не нужно обращаться к системным вызовам

"Hello, world" без компилятора

  • Напишем код на ассемблере, слинкуем и запустим
  • Постараемся максимально изобрести велосипед - сделать все своим кодом
  • printf нас точно не устроит
  • Нужно делать write в файловый дескриптор 1 (stdout)
  • Но библиотечный write - не функция
  • Это системный вызов сделаем его сразу
  • Будет три версии: linux x86 через nasm, linux на x86 через gas, osx на arm через gas

Пример


section .data
    msg db "Hello, World!", 0xa
    len equ $ - msg
section .text
    global _start
_start:
    mov rax, 1
    mov rdi, 1
    mov rsi, msg
    mov rdx, len
    syscall

    mov rax, 60
    xor rdi, rdi
    syscall

Пример


nasm -f elf64 -o hello.o hello.asm
ld -o hello hello.o
./hello

Пример

.section .data
    message: .ascii "Hello, world!\n"
    length: .set length, . - message
.section .text
.global _start
_start:
    movl $4, %eax
    movl $1, %ebx
    movl $message, %ecx
    movl $length, %edx
    int $0x80

    movl $1, %eax
    xorl %ebx, %ebx
    int $0x80

gcc -m32 -nostdlib hello.s -o hello

Модель процессора x86

  • Существуют RISC и CISC-архитектуры
  • RISC - логичные регулярные простые инструкции
  • Упрощает реализацию в железе
  • Но более длинный по числу инструкций код
  • В CISC более увесистые и выразительные инструкции
  • Более гетерогенное распределение ролей регистров
  • x86 - CISC-архитектуры
  • С незначительными RISC-тенденциями
