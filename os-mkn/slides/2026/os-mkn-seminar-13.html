Логи: постановка задачи

  • В мире баз данных есть понятие ACID
  • Atomicity, Consistency. Isolation, Durability
  • Частично он проецируется и на файловые системы
  • Как минимум - на метаданные

Логи: постановка задачи

  • Atomicity - группа операций или выполняется целиком, или не выполняется целиком
  • Durability - завершившаяся операция реально имеет эффект
  • Любой рестарт после завершенной операции не потеряет ее результаты
  • Durability вступает в противоречие с производительностью
  • Durability в наивной реализации приводит к "маленьким" операциям

Логи: постановка задачи

  • Isolation можно разделить на две части
  • "Снаружи" - не хотим видеть результаты незаверненной транзакции
  • "Изнутри" - можем захлтеть зафиксировать состояние базы на момент начала транзакции
  • И не видеть даже завершенные после этого транзакции

Логи: постановка задачи

  • Consistency - необходимость соблюдения инвариантов
  • В общем случае инварианты могут определяться в терминах системы в целом
  • Основная сложность - в изолиции параллельно работающих транзакций
  • Идеальная цель - сериализуемость транзакций
  • И блокирование несериализуемых последовательностей

Логи

  • В контексте файловых систем хотим решить Atomicity+Durability
  • Не жертвуя производительностью
  • Будем писать лог событий
  • Ради Durability обязаны писать операции на диск
  • Обновление диска через append помогает избегать позиционирования
  • Фоновый процесс переносит обновления в штатное место
  • До переноса данные могут браться из кеша

Undo-лог

  • Будем писать в лог старые значения того, что обновляем
  • И маркеры начала и конца транзакции
  • До написания маркера конца транзакции транзакцию надо закомитить
  • При рестарте ищем в логе транзации без маркера конца
  • Для них возвращаем старые значения

Undo-лог

  • Ключевая проблема Undo-лога - необходимость делать коммит
  • Это очень плохо для производительности
  • Теоретически не так критично для транзакций с большим числом обновлений
  • Но это не случай файловых систем

Redo-лог

  • Redo-лог фиксирует новые значения
  • Завершение транзакции не требует коммита
  • При рестарте потенциально может потребоваться воспроизведение обновлений всех закомиченных транзакций
  • Можно добавить запись по факту завершения синхронизации транзакции

Checkpoint

  • Не хотим хранить вечно растущие логи
  • Checkpoint - механизм определения ненужного префикса лога
  • Не останавливая поток операций
  • Отличается для Undo и Redo

Checkpoint для Undo

  • Пишем в лог START CHECKPOINT
  • И ID незавершенных транзакций
  • По мере их завершения пишем END CHECKPOINT
  • Участок до последнего завершенного checkpoint-а становится ненужным

Checkpoint для Redo

  • Пишем в лог START CHECKPOINT
  • И ID незавершенный транзакций
  • Активизируем запись на диск
  • По мере завершения пишем END CHECKPOINT
  • Участок до начала первой незавершенной транзакции становится ненужным

Смежные идеи и проблемы

  • Redo-лог можно передавать на другой узел - получится переливка данных
  • Можно организовать горячий бекап
  • Есть проблема вытеснения кеша до того, как обновленные данные записались в законное место
  • Можно организовать поиск по логу - и сделать лог хранилищем данных

Сети в контексте ОС

  • Сетевые возможности в контексте ОС можно понимать как еще один вариант межпроцессоного взаимодействия
  • Только процессы могут жить на разных узлах
  • Архитектура предполагает разнообразие протоколов на разных уровнях
  • На практике некоторые протоколы доминируют
  • И в силу этого особо важны

Общие принципы

  • Коммутация пакетов (не каналов)
  • Уровни протоколов
  • Нижний уровень - "физический"
  • Доступ к сетевой среде
  • Витая пара, коаксиальный кабель, WiFi

Общие принципы

  • Уровень выше - локальные сети
  • Пример: Ethernet
  • Адресация - MAC-адрес
  • Разделяемая физическая среда
  • Трафик между узлами физически доступен третьм узлам в той же сети

Общие принципы

  • Уровень выше - передача данных между локальными сетями
  • IP-адреса
  • Маршрутизация
  • Проблемы доставки: потери пакетов, пропажа пакетов, дублирование, наружение порядка

IP и UDP

  • IP - глобальный транспортный протокол
  • Старается обеспечить доставку пакетов от узла к узлу - через промежуточные сети
  • Адресация - от узла к узлу
  • Не учитывает приложения
  • UDP - легкая надстройка над IP
  • Появляется понятие "port" - для адресации приложения внутри хоста

TCP

  • TCP - дорогая надстройка над IP
  • Обеспечивает поток данных: файло-подобный интерфейс к сети
  • За кулисами - нумерация пакетов, перепосылки при необходимости
  • Основа клиент-серверной архитектуры
  • ВАЖНО: гарантии TCP работают в пределах соединения и в рамках сеанса

Реализация в ОС

  • IP и ниже - реализуется в ядре Linux
  • TCP и UDP - обычно управляется приложением
  • Выше TCP и UDP - прикладные протоколы
  • Для приложения используется абстракция сокета

Реализация в ОС

  • Сокет бывает клиентским и серверным
  • Сокет привязан к точке доступа
  • Точка доступа - в первую очередь сетевой интерфейс
  • Или что-то синтетически-виртуальное

Серверный сокет

  • Жизненный цикл: создание, bind, listen, accept
  • После accept получаем клиентский сокет у себя
  • Соединенный с удаленным клиентским сокетом
  • И возможность вернуть в accept
  • Вместе с необходимостью общаться по клиентскому сокету

Пример


    int server_fd;

    if ((server_fd = socket(AF_INET, SOCK_STREAM, 0)) < 0) {
        perror("socket failed");
        exit(EXIT_FAILURE);

    // Forcefully attaching socket to the port 8080
    if (bind(server_fd, (struct sockaddr*)&address,
             sizeof(address))
        < 0) {

        exit(EXIT_FAILURE);
    }

Параллелизм

  • Острее всего на стороне сервера
  • Для клиента тоже бывает актуально
  • Лобовое решение - fork
  • Дорогое и плохо масштабируемое
  • Можно лечить пулингом
  • Но есть неудобства с разными адресными пространствами

Параллелизм

  • Альтернатива - нити
  • В терминах Linux это системный вызов clone
  • В библиотеках это прорастает в виде концепции Thread
  • Воспринимается как дихотомия нить/процесс
  • На уровне ОС вариантов больше, чем 2

Параллелизм

  • В Linux нити по стоимости сопоставимы с процессами
  • Но их легче пулить
  • Оборотная сторона: нить сложнее аккуратно завершать
  • На нити нет аналога kill -9 - по принципиальным причинам

Параллелизм

  • Еще одна альтернатива - асинхронная работа
  • Базируется на системном вызове select
  • В цикле наблюдаем за изменением состоянием дескрипторов
  • Обрабатываем тех, у кого изменилось состояние
  • Нужна хороша языково-библиотечная поддержка

Контейнерная виртуализация

  • В основе - три механизма
  • namespace - механизм "изменения сознания" процессов
  • control group - механизм управления ресурсами
  • union fs - интересная файловая система
  • В чистом и беспримесном виде это - достояние Linux

namespace

  • В Linux есть 8 типов namespace-в
  • Каждый процесс принадлежит одному namespace-у каждого типа
  • Типу namespace-а соответсвует подсистема ядра, в отношении которой элементы ns "изменяют сознание"
  • Простейший тип - UTS namespace
  • Он про представление процесса о именах хоста и доменном имени
  • Изначально хост и доменное имя - глобальные свойства системы
  • Но для группы процессов можно изменить представление о имени хоста и доменном имени

Создание нового namespace-а

  • Есть два способа создать ns
  • unshare - создает ns заданных типов и пересекает в них текущий процесс
  • Есть одноименная утилита: sudo unshare --uts /bin/bash
  • clone - создание нового процесса в новых ns
  • fork создает ребенка в ns родителя

Другие типы

  • pid ns - представление о процессах
  • mount ns - представление о vfs и устройствах
  • user ns - представление о пользователях
  • network ns - представление о сетях
  • ipc ns - представление о interprocess communication
  • time ns - представление о времени
  • cgroup ns - представление о control group

Union FS

  • Механизм монтирования с fallback-ом
  • Ищем путь в одной файловой системе
  • Если не нашли - ищем в другой
  • На этом строится механизм слоев
  • Экономим дисковую память на контейнерах со сначительным пересечением по содержимому файлов

cgroup

  • cgroup - механизм ограничения ресурсов
  • Можно ограничивать cpu, память и т.п.
  • Работает по принциту файловой системы
  • cgtoup ns задает корень файловой системы для ns
